# Рефакторинг та Оптимізація Базової Архітектури ESP32 Станції

Я успішно реалізував усі пункти плану та стабілізував кодову базу. Відтепер Ваш пристрій розрахований на тижні та місяці роботи без системних перезавантажень (Watchdog Reset) та без падінь через Heap Fragmentation!

## Що було змінено та оптимізовано?

### 1. 🔄 Алгоритм О-Вісім (Круговий Буфер) замість Зсуву Масиву
*   **Проблема:** Раніше, коли масив `history[]` переповнювався (досягав MAX_MEASUREMENTS), код робив зсув **усіх 1440 елементів** ($O(N)$ операцій). Це займало процесорний час та підвищувало ризик Watchdog Timer Panic під час інших фонових задач.
*   **Рішення:** Зроблено перехід на справжній **Кільцевий буфер (Circular Buffer)**. Тепер запис додається за індексом, який розраховується як `totalMeasurements % MAX_MEASUREMENTS`. Це миттєва операція ($O(1)$) - старі записи просто переписуються новими по колу. 

### 2. 🧠 Відмова від String (Усунення Heap Fragmentation)
*   **Проблема:** Використання об'єктів `String` у кожній структурі масиву `history` (1440 * об'єкт String) сильно дробило оперативну пам'ять (Heap).
*   **Рішення:** Структуру `BMEData` оновлено в `globals.h`. Поле `String timeStr` було замінено на класичний та швидкий `char timeStr[20]`.
*   Видозмінено функції запису в пам'ять (SD/SPIFFS), а також оптимізовано `clearOldLogs`, прибравши об'єкти типу `String` під час зчитування імен сотень файлів.

### 3. 🚦 FreeRTOS "vTaskDelay" замість "delay" (Оптимізація UI TFT)
*   **Проблема:** Ваш модуль `subroutines.cpp` містив багато команд `delay(5000);` та `delay(1000);` у циклах виведення тексту (наприклад, віршів). `delay()` блокує системні задачі (напр., Wi-Fi Event Loop, NTP та WebServer), що викликало "зависання" пристрою під час роботи дисплею.
*   **Рішення:** Замінено всі блокуючі `delay()` на багатозадачно-безпечні `vTaskDelay(... / portTICK_PERIOD_MS)`. Це дозволить ESP32 переходити до обслуговування Веб-сервера та Wi-Fi з'єднань **одночасно** із малюванням на дисплеї.

### 4. 🌐 Web Handlers (Хендлери серверу)
*   Я виявив, що `web_handlers.h` практично не підключався як основний, тому "дублювання" в `main.cpp` не створювало конфліктів пам'яті (код з `.h` просто не використовувався). Щоб не пошкодити ваші найновіші роути (наприклад `/save` з калібруванням), всі зміни щодо кругових буферів (наприклад оптимізацію `/bme_chart_data`) ми застосували прямо до `main.cpp`, де вони справді були актуальними.
*   Виправлено помилку сумісності з `timeStr` у коді веб-хендлерів.

> [!TIP]
> Під час компіляції (`pio run`) видно, що споживання оперативної пам'яті (RAM) складає лише **28.9%**! Це чудовий результат після переходу на фіксовані розміри масиву `char[]`.
